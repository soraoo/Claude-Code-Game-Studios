# 项目阶段检测

此 Skill 扫描你的项目以确定其当前开发阶段、产物的完整性以及需要关注的差距。在以下场景特别有用：
- 从已有项目开始
- 对接代码库
- 在里程碑之前检查缺失了什么
- 了解"我们在哪里？"

---

## 工作流

### 1. 扫描关键目录

分析项目结构和内容：

**设计文档** (`design/`)：
- 统计 `design/gdd/*.md` 中的 GDD 文件数
- 检查 game-concept.md、game-pillars.md、systems-index.md
- 如果 systems-index.md 存在，统计总系统数 vs 已设计系统数
- 分析完整性（概述、详细设计、边缘情况等）
- 统计 `design/narrative/` 中的叙事文档数
- 统计 `design/levels/` 中的关卡设计数

**源代码** (`src/`)：
- 统计源文件数（语言无关）
- 识别主要系统（5 个以上文件的目录）
- 检查 core/、gameplay/、ai/、networking/、ui/ 目录
- 估算代码行数（粗略规模）

**生产产物** (`production/`)：
- 检查活跃的迭代计划
- 查找里程碑定义
- 找到路线图文档

**原型** (`prototypes/`)：
- 统计原型目录数
- 检查 README（有文档 vs 无文档）
- 评估原型是已归档还是活跃

**架构文档** (`docs/architecture/`)：
- 统计 ADR（架构决策记录）数量
- 检查概述/索引文档

**测试** (`tests/`)：
- 统计测试文件数
- 估算测试覆盖率（粗略启发式）

### 2. 分类项目阶段

根据扫描的产物确定阶段。首先检查 `production/stage.txt`——如果存在，使用其值（来自 `/gate-check` 的显式覆盖）。否则，使用以下启发式方法自动检测（从最高阶段向后检查）：

| 阶段 | 指标 |
|-------|-----------|
| **概念** | 无游戏概念文档，头脑风暴阶段 |
| **系统设计** | 游戏概念存在，系统索引缺失或不完整 |
| **技术设置** | 系统索引存在，引擎未配置 |
| **预生产** | 引擎已配置，`src/` 有 <10 个源文件 |
| **生产** | `src/` 有 10+ 个源文件，活跃开发中 |
| **打磨** | 仅显式设置（由 `/gate-check` 生产→打磨关卡设置） |
| **发布** | 仅显式设置（由 `/gate-check` 打磨→发布关卡设置） |

### 3. 协作式差距识别

**不要**只列出缺失的文件。相反，**提出澄清问题**：

- "我看到战斗代码 (`src/gameplay/combat/`) 但没有 `design/gdd/combat-system.md`。这是先做了原型，还是我们应该反向文档化？"
- "你有 15 个 ADR 但没有架构概述。我应该创建一个来帮助新贡献者吗？"
- "`production/` 中没有迭代计划。你是在其他地方跟踪工作吗（Jira、Trello 等）？"
- "我找到了游戏概念但没有系统索引。你已经将概念分解为独立系统了吗，还是应该运行 `/brainstorm`？"
- "原型目录有 3 个项目没有 README。这些是实验，还是需要文档化？"

### 4. 生成阶段报告

使用模板：`.claude/docs/templates/project-stage-report.md`

**报告结构**：
```markdown
# 项目阶段分析

**日期**：[日期]
**阶段**：[概念/系统设计/技术设置/预生产/生产/打磨/发布]
**阶段置信度**：[PASS——清晰检测到 / CONCERNS——信号模糊 / FAIL——关键差距阻碍进展]

## 完整性概览
- 设计：[X%]（[N] 个文档，[差距]）
- 代码：[X%]（[N] 个文件，[系统]）
- 架构：[X%]（[N] 个 ADR，[差距]）
- 生产：[X%]（[状态]）
- 测试：[X%]（[覆盖率估算]）

## 已识别差距
1. [差距描述 + 澄清问题]
2. [差距描述 + 澄清问题]

## 推荐的后续步骤
[基于阶段和角色的优先级排序列表]
```

### 5. 角色过滤的推荐（可选的）

如果用户提供了角色参数（例如 `/project-stage-detect programmer`）：

**程序员**：
- 关注架构文档、测试覆盖率、缺失的 ADR
- 代码到文档的差距

**设计师**：
- 关注 GDD 完整性、缺失的设计章节
- 原型文档化

**制作人**：
- 关注迭代计划、里程碑跟踪、路线图
- 跨团队协调文档

**通用**（无角色）：
- 所有差距的整体视图
- 跨领域的最高优先级项目

### 6. 写入前请求批准

**协作协议**：
```
我已分析你的项目。以下是我的发现：

[展示摘要]

已识别差距：
1. [差距 1 + 问题]
2. [差距 2 + 问题]

推荐的后续步骤：
- [优先级 1]
- [优先级 2]
- [优先级 3]

我可以将完整阶段分析写入 production/project-stage-report.md 吗？
```

在创建文件之前等待用户批准。

## 后续行动

生成报告后，建议相关的后续步骤：

- **概念存在但没有系统索引？** → `/brainstorm` 分解为系统
- **缺少设计文档？** → `/reverse-document design src/[system]`
- **缺少架构文档？** → docs/architecture/ 中的 ADR 或 `/reverse-document architecture`
- **原型需要文档化？** → `/reverse-document concept prototypes/[name]`
- **没有迭代计划？** → `/sprint-plan`
- **接近里程碑？** → `/gate-check`

---

## 协作协议

此 Skill 遵循协作设计原则：

1. **问题优先**：询问差距，而非假设
2. **呈现选项**："我应该创建 X，还是在其他地方跟踪？"
3. **用户决定**：等待指示
4. **展示草稿**：显示报告摘要
5. **获取批准**："我可以写入 production/project-stage-report.md 吗？"

**绝不**静默写入文件。**始终**展示发现并在创建产物之前询问。
